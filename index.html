What is a repository?

A repository (repo) is a folder that Git tracks — every file, every change, every version, forever.

Think of a repository like a Google Doc for your entire codebase — but instead of tracking one file, it tracks thousands, and instead of one author, it handles an entire team working simultaneously without overwriting each other.

Anatomy of a repository

PartWhat it is
.git/Hidden folder where Git stores the full history of every change — never delete this
.gitignoreList of files Git should ignore (e.g. .env, node_modules)
README.mdThe front page of your repo — always keep it up to date
main branchThe stable, production-ready version of the code
Feature branchesWhere your work lives until it's reviewed and approved

Cloning a repo

# Clone via HTTPS (recommended for interns) git clone https://github.com/jsdevbrains/project-name.git # Move into the project cd project-name # See what's inside ls -la # Check Git status — always do this before anything git status

The three states of your files

📝

Modified

You've changed a file but haven't told Git about it yet. Run git status to see these.

📋

Staged

You've run git add — Git knows about the change and it's ready to be committed.

✅

Committed

The change is saved in Git's history permanently. Every commit gets a unique ID.

# See what changed git status # Stage specific file git add src/components/Button.jsx # Stage everything (use carefully) git add . # Commit with a message git commit -m "Add primary button component with hover states" # See commit history git log --oneline
✓ Good commit messages
  • Add login form validation
  • Fix null pointer in user service
  • Refactor auth middleware for clarity
  • Update README with API endpoints
✗ Bad commit messages
  • fix
  • changes
  • stuff
  • asdasd

Feature branches

Branches let you work in isolation — your changes never touch anyone else's code until they're reviewed and approved.

⚠️ Never commit directly to main. Every piece of work — no matter how small — gets its own branch. This is non-negotiable at JSDEVBRAINS.

Handling merge conflicts

A merge conflict happens when two people edit the same line. Git can't choose — you have to decide.

# Git marks conflicts like this in the file: <<<<<<< HEAD Your version of the code ======= Their version of the code >>>>>>> origin/main # Steps to resolve: # 1. Open the file, delete the markers, keep what's correct # 2. Stage the resolved file git add src/file-with-conflict.js # 3. Continue git rebase --continue
💡 Always ask your supervisor before force-pushing (git push --force). In most cases, it's the wrong call.

PR rules at JSDEVBRAINS

✓ Good PR habits
  • Keep PRs small and focused (one task)
  • Use Draft PR for work in progress
  • Add screenshots for UI changes
  • Respond to every review comment
  • Wait for approval before merging
✗ PR anti-patterns
  • Opening a 2000-line PR nobody wants to review
  • Merging your own PR without review
  • Ignoring review comments silently
  • Pushing to a PR after it's approved
  • Leaving PRs open for more than a week

Security guidelines

One leaked credential can compromise a client's system. These rules protect you, the team, and our clients.

🚨 Never commit secrets. Not once. Not "just for testing." Not "I'll remove it later." GitHub history is permanent — even after deletion, secrets can be found.

What never goes into Git

✗ Never commit these
  • Passwords and database credentials
  • API keys (Stripe, AWS, Twilio, Google, etc.)
  • JWT secrets or private keys (.pem, .key files)
  • Production environment variables
  • Personal access tokens
  • Anything inside .env or .env.local

The correct approach — environment variables

# .env (never committed) DATABASE_URL=postgresql://user:password@localhost/mydb STRIPE_SECRET_KEY=sk_live_abc123... JWT_SECRET=super-secret-value # In your code — load from environment, never hardcode const db = process.env.DATABASE_URL const stripe = new Stripe(process.env.STRIPE_SECRET_KEY)

Your .gitignore — non-negotiable entries

# Environment files .env .env.local .env.*.local .env.production # Keys and certificates *.pem *.key *.p12 *_rsa *_secret* # Dependencies (never commit these) node_modules/ .next/ dist/ build/ # OS junk .DS_Store Thumbs.db

What to do if you accidentally commit a secret

1

Tell your supervisor immediately

Don't try to fix it silently. Speed matters — the secret needs to be rotated.

2

Revoke the secret

Go to the service (AWS, Stripe, etc.) and invalidate the key right now, before anything else.

3

Remove from history

Use git filter-branch or BFG Repo Cleaner with supervisor guidance.

4

Force push the cleaned history

Requires supervisor approval — this rewrites history for everyone.

Scenario: "I pushed to main by accident"

🚨 Stop. Do not try to fix it alone. Tell your supervisor immediately. They will guide the recovery — trying to self-fix often makes it worse.

Scenario: "My branch is behind by 50 commits"

# Rebase your branch onto latest main (preferred) git fetch origin git rebase origin/main # If there are conflicts, resolve them, then: git add . git rebase --continue # If it gets too messy — abort and ask for help git rebase --abort

Scenario: "I need to undo my last commit"

# Undo last commit but keep changes staged git reset --soft HEAD~1 # Undo last commit and unstage changes (files still changed) git reset HEAD~1 # ⚠️ Nuclear option — destroys changes permanently git reset --hard HEAD~1
⚠️ Never use git reset --hard if your commits are already pushed to a shared branch. You will lose work and cause problems for your team.

The engineering mindset — 7 steps when stuck

1

Understand the problem

Can you explain it in plain words? If not, you don't understand it yet.

2

Read the error message fully

The answer is usually there. Read every line before Googling.

3

Check the documentation

Official docs exist for a reason. Read them before Stack Overflow.

4

Search for the exact error

Copy the error message, paste it in Google. Someone has hit this before.

5

Try something and observe

A failed attempt teaches you more than waiting for certainty.

6

Ask for help — with context

Tell your supervisor: what you tried, what you expected, what happened. Not just "it doesn't work".

7

Document what you learned

Post it in GitHub Discussions. Future interns will find it.

Test yourself

Quick-fire questions to check your understanding. No pressure — use the answers to fill gaps.
